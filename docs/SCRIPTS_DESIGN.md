# 脚本系统设计 (SCRIPTS_DESIGN)

本项目通过 `scripts/` 目录集中管理所有运维、启动及辅助工具脚本，旨在降低操作复杂度并实现流程标准化。

## 1. 脚本目录结构

```text
scripts/
├── load_env.sh              # 核心脚本：加载全局环境变量到当前 Shell
├── Antigravity-Manage/      # AI Agent 专用的启动与管理逻辑
└── Docker-Manage/           # 通用 Docker 容器运维工具（重启、清理、日志查看等）
```

## 2. 核心机制：环境变量加载 (load_env.sh)

由于 `docker compose` 及许多管理脚本依赖根目录下的 `.env` 配置，系统提供了统一的加载机制。

- **实现逻辑**: 使用 `set -a` (allexport) 指令，将 `.env` 文件中的所有赋值语句自动标记为环境变量导出（Export）。
- **执行规范**: 必须使用 `source` 或 `.` 命令执行，以便变量作用于当前的 Shell 会话。
  ```bash
  source scripts/load_env.sh
  ```
- **自动化集成**: 在 `Docker-Manage` 等项目的入口脚本中，应优先调用此加载逻辑，确保 `docker compose` 能够正确识别 `${VARIABLE}` 语法。

## 3. 脚本编写规范

- **路径无关性**: 脚本内部应通过 `$(dirname "${BASH_SOURCE[0]}")` 获取自身路径，确保在任何工作目录下执行均能正确找到项目根目录。
- **幂等性**: 脚本应支持重复执行而不产生副作用。
- **错误处理**: 关键步骤应检查返回值，并在失败时提供清晰的错误提示信息。
- **注释要求**: 脚本头部必须包含功能描述及使用示例。

## 4. 管理与扩展

- **权限管理**: 所有 `.sh` 脚本在提交前应确保具备可执行权限 (`chmod +x`)。
- **解耦原则**: 避免在脚本中硬编码个人路径，应通过 `load_env.sh` 提供的 `MNG_HOME`, `USER_HOME` 等变量进行引用。
