# 用户、权限与环境变量规范 (USER_PERMISSIONS_ENV)

本文档定义了管理工程在用户身份控制、文件系统权限以及环境变量管理上的核心准则，确保系统在安全性与可维护性之间达到平衡。

## 1. 用户管理 (Users)

为了保障宿主机安全，所有容器内进程原则上应以非 root 用户身份运行。

- **默认身份**: 容器内统一映射 UID/GID 为 `1000:1000`（即宿主机当前用户 `nick`）。
- **UID/GID 注入**: 
    - 在 `ops-gitea` 等需要特定权限的服务中，通过环境变量 `USER_UID` 和 `USER_GID` 动态注入。
    - 在 Dockerfile 或 `entrypoint` 中预先设置挂载目录的所有权。

## 2. 权限规范 (Permissions)

严格遵守“最小权限原则”，禁止对任何挂载目录执行 `chmod 777`。

### 2.1 宿主机挂载点
- **目录权限**: 统一设为 `755` (drwxr-xr-x)。
- **文件权限**: 不低于 `640` (-rw-r-----)。

### 2.2 特殊敏感文件
- **证书文件**: `acme.json` 等证书相关文件必须设置为 `600` (-rw-------)。
- **SSH 密钥**: `link/github/` 下的公钥链接遵循普通文件权限，私钥禁止创建链接。

## 3. 环境变量管理 (Environment Variables)

环境变量分为“工程全局”、“层级/服务专用”及“敏感信息”三类。

### 3.1 存放路径
- **工程全局**: 位于根目录的 `.env` 文件。
- **服务专用**: 位于 `docker/env/` 目录下，按 `<层级>-<服务名>.env` 命名。

### 3.2 敏感数据保护
- **模板机制**: 每个 `.env` 文件必须对应一个 `.env.example` 模板。
    - `*.env.example`: 仅含键名，值为空。必须提交至 Git。
    - `*.env`: 包含真实敏感值（如数据库密码、API Token）。**严禁**提交至 Git，必须加入 `.gitignore`。
- **配置链接**: `link/env/` 存放宿主机 Shell 环境变量配置的符号链接，方便统一查看但不纳入版本库（敏感内容除外）。

## 4. 管理流程
1. **新增服务时**: 首先在 `docker/env/` 创建对应的 `.env.example`。
2. **部署新环境**: 参照 `.env.example` 复制并填写真实的 `.env` 文件。
3. **权限检查**: 在启动容器前，确保 `mounts/` 下对应的目录已预先创建且权限正确。
