# 测试规范文档 (TEST_SPECIFICATION)

本规范定义了 NickManage 系统的测试分层体系、自动化流程及质量准入标准，确保系统变更的可靠性与稳定性。

## 1. 测试分层体系 (Testing Layers)

系统测试按照“由内而外、由易到难”的原则分为三个核心层级。

### 1.1 基础健康巡检 (Liveness Check)
- **目标**: 确保所有核心容器（ops/share/dev/app）处于 `running` 状态且 API 基础路径通畅。
- **执行频率**: 每次系统启动、容器重启或配置变更后。
- **核心工具**: `test/ops_test.sh` (Step 1-3)。

### 1.2 业务集成测试 (Integration Testing)
- **目标**: 验证跨容器、跨层级的业务逻辑闭环。
- **验证点**:
    - **日志链路**: 业务日志 -> Vector -> Loki。
    - **认证授权**: Docker Proxy 对写操作的拦截。
    - **网络互通**: 业务层正常访问共享层的数据库与 AI 模型。
- **核心工具**: `test/demo_test.sh`。

### 1.3 稳定性与自愈测试 (Stability & Recovery)
- **目标**: 验证系统在极端或异常情况下的鲁棒性。
- **验证点**:
    - **僵尸进程检测**: `ops-monitor` 对挂死服务的自动检测与物理重启。
    - **数据备份恢复**: 定期自动进行备份文件的完整性校验。
- **核心工具**: `test/stability_test.sh` 及后续补充的备份测试脚本。

---

## 2. 自动化执行规范 (Automation)

### 2.1 本地手动触发
在进行关键配置修改后，开发者必须在 `NickManage` 根目录下执行：
```bash
# 验证底座层
bash test/ops_test.sh

# 验证演示项目闭环
bash test/demo_test.sh
```

### 2.2 CI/CD 自动化集成 (Gitea Actions)
- **提交触发**: 每次 `push` 代码至管理仓时，自动触发关键路径的健康巡检。
- **合并触发**: 合并至 `main` 分支前，必须通过全量稳定性验证。

---

## 3. 质量准入与准出标准 (Quality Gates)

| 阶段 | 准入标准 | 准出（通过）标准 |
| :--- | :--- | :--- |
| **底座部署** | `.env` 文件已配置且 `nms-bridge` 已创建。 | `ops_test.sh` 全绿通过，Loki 成功接收初始心跳日志。 |
| **项目创建** | 已通过 `create_project.sh` 生成对应容器配置。 | `demo_test.sh` 验证新容器生命周期与网络连通性。 |
| **稳定性演进** | 系统运行超过 72 小时。 | `stability_test.sh` 模拟场景下自愈耗时 < 60s。 |

---

## 4. 测试盲区处置
对于无法自动化验证的场景（如宿主机硬件故障、大规模断电），应定期执行人工巡检并记录在运维日志中。
